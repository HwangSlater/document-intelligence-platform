## 2. 객관적인 시장 위치 재평가

* **"상위권", "차별성" 표현에 대한 냉정한 분석**:
  * 단순 API 호출 래퍼 코드로 포트폴리오를 채운 다수 지원자에 비해, DDL 관계 매핑이나 `JdbcTemplate` 제어, 배치 임베딩, 조기 우회 로직을 적용한 점은 **"엔지니어링 문제 해결 시도가 돋보이는 지원자"**로 분류하기에 충분합니다.
  * 그러나 이것이 곧바로 "취업 프리패스"나 "압도적 우위"를 뜻하지는 않습니다. 실제 대용량 트래픽 인프라나 복잡한 웹 비즈니스 로직을 다루는 일반 백엔드 신입 지원자군과 1:1로 비교될 때, RAG 파이프라인의 단순 CRUD 성격은 한계가 있습니다. 
  * 따라서 "AI 기술을 많이 안다"가 아니라 **"AI 도구를 다루는 과정에서 발생한 백엔드 병목을 주도적으로 튜닝했다"**로 포지셔닝해야 비교 우위를 점할 수 있습니다.

---

## 3. 면접관 입장에서의 팩트 체크 및 톤 조절

1. **PgVectorStore 대신 JdbcTemplate을 사용한 이유**:
   * **과장된 설명**: "JPA나 Spring AI가 지원하지 않아서 직접 짰습니다." (❌ 감점)
   * **현실적이고 신뢰받는 설명**: "Spring AI의 기본 VectorStore는 메타데이터와 벡터를 JSON 단일 컬럼에 쏟아부어 테이블 간 관계 설정이 차단됩니다. 본 프로젝트에서는 `Document` - `Chunk` - `Embedding` 간의 라이프사이클 정합성을 강제하기 위해 `ON DELETE CASCADE` 외래키 제약조건과 `UNIQUE(chunk_id)` 무결성을 보장해야 했고, pgvector 쿼리를 온전히 통제하기 위해 JdbcTemplate 직접 제어 방식을 택했습니다."
2. **HNSW 사용 배제 사유**:
   * **신입 수준에 적절한 설명**: "HNSW는 Approximate Nearest Neighbor(ANN) 탐색이므로 정확도가 소폭 희생됩니다. 현재 데이터 규모(소량 문서 청크)에서는 Exact Search(Full Scan) 방식으로도 밀리초(ms) 단위 검색이 가능하며 재현율(Recall) 100%를 보장하므로, 굳이 인덱스 빌드 및 메모리 오버헤드를 안고 HNSW를 도입할 트레이드오프가 성립하지 않아 제외했습니다."
3. **RetrievalService의 확장성 설명**:
   * **과장된 설명**: "향후 Agent 단계와의 충돌을 막기 위해 분리했습니다." (❌ 비약으로 보임)
   * **현실적인 설명**: "유사 청크 조회 기능과 대화 생성 기능은 역할이 다릅니다. 검색 로직을 `RetrievalService`로 완전히 분격시켜 둠으로써, 검색 모듈의 느슨한 결합(Loose Coupling)을 확보하고 단독으로 재사용이 가능하도록 설계했습니다."
4. **TokenTextSplitter 트러블슈팅 강조 수준**:
   * "기본 빌더와 축약형 생성자 사용 시 내부 설정 수치상 maxNumChunks 한도가 1개로 제한되던 라이브러리 사용성 및 설정 튜닝 이슈"로 한정하여, 오픈소스 내부 코드를 직접 까보고 명시적인 생성자 파라미터 주입으로 해결한 **디버깅 프로세스 자체**를 강조하십시오.
5. **Batch Embedding의 강점 정의**:
   * "성능의 획기적 향상"이라는 모호한 표현 대신, **"네트워크 API 호출 횟수를 N분의 1로 줄여 OpenAI Rate Limit(RPM) 한도 도달 리스크를 차단하고, API Latency 구간을 최적화하기 위한 실무적인 비용/성능 세이빙 시도"**로 포지셔닝해야 설득력이 높습니다.

## 6. 면접용 1분 자기소개 & 3분 프로젝트 소개 스크립트

### **1분 버전 (자기소개 또는 프로젝트 요약)**
> "안녕하세요, AI 백엔드 지원자 OOO입니다. 
> 저는 프레임워크가 제공하는 추상화 API 뒤에 숨지 않고, 데이터와 데이터베이스 레벨을 직접 통제하여 문제를 해결하는 백엔드 개발자입니다. 
> 이번 '문서 지능 플랫폼' 프로젝트에서는 Spring AI의 기본 VectorStore 대신 pgvector 데이터베이스 스키마와 관계를 직접 모델링하고, `JdbcTemplate`을 이용해 코사인 거리 최적화 쿼리를 작성했습니다. 
> 또한, RAG 구현 시 발생할 수 있는 OpenAI API 호출 요금을 절감하기 위해 검색 결과 부재 시 LLM 호출을 건너뛰는 조기 우회 반환 로직을 설계했습니다. 
> 이처럼 비즈니스의 비용과 인프라의 동작 구조를 깊이 이해하고 튜닝하는 개발자가 되겠습니다. 감사합니다."

### **3분 버전 (상세 프로젝트 소개 요청 시)**
> "제가 개발한 '문서 지능 플랫폼'은 PDF 파일에서 텍스트를 추출하고 청킹한 뒤, 이를 벡터로 변환하여 질문에 답변하는 RAG QA 백엔드 플랫폼입니다.
> 
> 프로젝트를 진행하며 세 가지 핵심 엔지니어링 문제를 정의하고 해결했습니다.
> 
> 첫째는 **데이터 정합성과 무결성의 확보**였습니다. Spring AI의 PgVectorStore는 메타데이터와 벡터를 단일 JSON 컬럼에 직렬화하여 적재하므로 테이블 간 관계나 인덱스 통제가 불가능했습니다. 이를 해결하기 위해 Documents, Chunks, Embeddings 테이블을 직접 정규화 모델링하여 `ON DELETE CASCADE`와 `UNIQUE` 제약을 직접 걸고, `JdbcTemplate` 다이렉트 쿼리로 통제했습니다.
> 
> 둘째는 **API 비용 및 지연시간 최적화**였습니다. 검색 질의 수행 시 매칭된 관련 문서 조각이 없을 때는 OpenAI ChatModel API를 호출하지 않고 서비스 단에서 우회 답변을 즉시 반환하도록 설계했습니다. 또한 코사인 유사도 점수 변환 정렬이 아닌 pgvector 인덱스 실행 경로를 보장하기 위해 코사인 거리 오름차순인 `ORDER BY e.embedding <=> ? ASC`로 쿼리를 최적화했습니다.
> 
> 셋째는 **라이브러리 매개변수 디버깅**이었습니다. TokenTextSplitter 사용 시 모든 텍스트가 1개의 청크로 병합되는 이슈가 발생했습니다. 오픈소스 코드를 파악한 결과, 내부 기본 빌더가 maxNumChunks 한도를 1개로 제한하는 기본 설정 값을 갖는 것을 발견하고, 커스텀 생성자를 주입해 한도를 해제하고 청킹을 정상화했습니다.
> 
> 결론적으로, 단순히 라이브러리를 가져다 쓰기만 한 것이 아니라, 동작 원리를 분석하고 DB 레벨의 쿼리를 튜닝하여 RAG 파이프라인의 백엔드 병목을 정교하게 제어한 경험을 축적했습니다. 이상입니다."

## 7. 면접관이 가장 좋아할 스토리 TOP 5 (Troubleshooting)

1. **`TokenTextSplitter` 파라미터 디버깅 스토리**:
   * *주제*: "라이브러리가 안 되면 내부 소스를 까본다."
   * *내용*: 청킹 개수가 1개에 머무는 이상 현상을 감지하고 오픈소스 내부 구현을 분석하여 기본 설정 제한을 밝혀내어 해결한 집요함.
2. **`AssistantMessage` 타입 호환성 수정 스토리**:
   * *주제*: "마일스톤 버전의 버전 호환성 트러블슈팅."
   * *내용*: Spring AI 1.0.0-M6 마일스톤 버전에서 `AssistantMessage`에서 텍스트를 가져올 때 `getContent()`가 아닌 `getText()`를 호출해야 컴파일이 통과되는 사양을 API 사양 서를 뒤져 해결한 경험.
3. **`JdbcTemplate`과 pgvector의 바인딩 캐스팅 해결 스토리**:
   * *주제*: "프레임워크 없이 pgvector 수동 바인딩하기."
   * *내용*: `float[]` 타입을 다이렉트로 매핑할 때 생기는 캐스팅 에러를 `Arrays.toString()`을 활용한 pgvector 표준 포맷 스트링 바인딩(`?::vector`)으로 안정화한 스토리.
4. **Fast-path 조기 우회 반환 도입 스토리**:
   * *주제*: "실무 비용을 아끼는 방어적 아키텍처."
   * *내용*: 검색 결과가 없는 상황에서 의미 없이 호출되는 LLM API를 차단하여 크레딧 요금 낭비를 막고 사용자 응답 속도를 개선한 흐름.
5. **H2 DDL Swallowing 설계 스토리**:
   * *주제*: "로컬/테스트 환경의 빌드 안정성을 지키는 우회로."
   * *내용*: pgvector 확장 모듈이 없는 H2 인메모리 테스트 환경에서 빌드가 깨지지 않도록 `@PostConstruct` 내 DDL 예외를 안전하게 가두어 격리 단위 테스트 빌드를 성공시킨 이야기.
