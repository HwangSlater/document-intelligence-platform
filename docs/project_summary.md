# 📘 Document Intelligence Platform 종합 분석 및 면접 준비 가이드 (Project Summary)

본 문서는 **Document Intelligence Platform** 프로젝트의 핵심 설계 사상, 구현 내역, 검증 결과 및 문제 해결 과정을 총망라한 종합 요약 문서입니다. 기술 면접에서 면접관을 설득하고 예상 질문을 방어하기 위한 백엔드 엔지니어링 가이드라인으로 활용할 수 있습니다.

---

## 1. 프로젝트 개요 & 비즈니스 목표

> **"단순히 문서를 보관하는 저장소를 넘어, 문서의 지식을 체계적으로 추출하고 이해하는 지능형 플랫폼"**

시중의 대다수 RAG(Retrieval-Augmented Generation) 튜토리얼은 고수준 AI 프레임워크가 제공하는 추상화 API(예: Spring AI 내장 VectorStore)를 사용하여 단순히 PDF를 적재하고 검색하는 수준에 머무릅니다. 
본 프로젝트는 **"추상화 뒤에 숨지 않고 영속성 계층과 데이터 흐름을 직접 통제한다"**는 설계 원칙 하에 아래와 같은 비즈니스 및 엔지니어링 목표를 달성했습니다.

1. **데이터 영속성 및 관계의 명확성 확보**: RDB 정규화 스키마 모델링과 네이티브 pgvector 연동을 손수 처리하여 데이터 정합성 보장.
2. **지연 시간 및 API 호출 최적화**: 구간별 지연 시간 계측을 통해 병목 지점(외부 LLM API)을 규명하고, Batch 처리 및 Fast-path Bypass로 인프라 비용 방어.
3. **수치 기반의 객관적 검증**: 9개의 실제 PDF 전문 문서와 1,084개의 청크를 적재하고, 20개의 검증용 골드셋 질문을 순회하며 검색 및 생성 결과의 정합성을 수동 평가하여 신뢰도 확보.

## 2. 전체 시스템 및 데이터 파이프라인 흐름

시스템은 크게 **[문서 적재 및 임베딩 파이프라인]**과 **[유사도 검색 및 RAG Q&A 파이프라인]**으로 분리되어 독립적으로 동작합니다.

### 🔄 데이터 적재 파이프라인 (Document Ingestion)
1. **PDF 업로드**: 클라이언트가 PDF 파일을 전송하면 `uploads/` 스토리지에 물리적으로 저장합니다. (`Document` 엔티티 생성: `UPLOADED` 상태)
2. **텍스트 추출**: **Apache Tika** 파서를 기동하여 PDF 본문과 메타데이터를 추출합니다. (`Document` 상태: `PARSED`로 전이)
3. **토큰 기반 청킹**: Spring AI의 `TokenTextSplitter`를 사용하여 추출된 원본 텍스트를 청크로 쪼갭니다. (**청크 크기: 500 토큰, 오버랩: 100자**) 분할된 조각은 `chunks` 테이블에 영속화됩니다.
4. **배치 임베딩**: 임베딩 생성 시, 문서 내 모든 청크의 텍스트를 수집해 **OpenAI `text-embedding-3-small` API로 단 1회 Batch 호출**하여 1536차원 벡터를 생성합니다.
5. **영속화**: `EmbeddingStore`가 `JdbcTemplate` 네이티브 SQL을 통해 기존 임베딩을 삭제(멱등성 보장)한 후, pgvector `VECTOR(1536)` 타입 컬럼에 벡터 데이터를 적재합니다.

### 🔍 유사도 검색 & RAG Q&A 파이프라인 (Retrieval & Generation)
1. **질문 접수**: 사용자가 질문을 던지면 `RagService`가 호출됩니다.
2. **질문 임베딩**: `RetrievalService`가 사용자의 질문 텍스트를 OpenAI Embedding API를 사용해 벡터값으로 변환합니다.
3. **유사도 검색 (Retrieval)**: PostgreSQL pgvector 코사인 거리 연산자 `<=>`를 사용하여 질문 벡터와 가장 가까운 **Top-5 청크**를 조회합니다.
4. **조기 우회 (Fast-path Bypass)**: 
   * 만약 DB에서 검색된 청크가 전혀 없으면(`chunks.isEmpty()`), 고비용의 외부 LLM 호출을 건너뛰고 서비스 단에서 즉각 `"제공된 문서 내에 해당 질문에 답변할 수 있는 관련 정보가 존재하지 않습니다."`라는 고정 답변을 반환합니다.
5. **RAG 프롬프트 빌딩 & LLM 호출**: 
   * 검색된 문맥이 존재하면 시스템 지침, 검색 문맥(Context), 질문(Question)을 하나의 템플릿에 결합하여 단일 Prompt를 만들고, OpenAI `gpt-4o` 모델에 질의하여 정합성 높은 최종 답변을 획득합니다.

## 3. 핵심 엔지니어링 의사결정 (Core Highlights)

### 💾 3단계 데이터베이스 정규화 모델링
Spring AI의 기본 제공 테이블(`vector_store`)은 단일 테이블 내에 JSON 형식으로 메타데이터와 벡터를 직렬화하여 채워 넣기 때문에, CASCADE 옵션을 활용한 관계 삭제나 관계형 쿼리 최적화가 불가능합니다.
본 프로젝트는 이를 정규화하여 관리합니다.

* **`documents` (1)**: 파일의 물리적 정보 및 상태 관리 (`ON DELETE CASCADE`의 근원)
* **`chunks` (N)**: 쪼개진 텍스트 데이터 보관 및 순서 관리
* **`embeddings` (1:1 with chunk)**: `chunk_id` UNIQUE 제약 조건을 통한 1:1 관계 보장, pgvector 데이터 타입 바인딩 및 `idx_embeddings_chunk_model` 인덱스 최적화

### ⚡ 코사인 유사도 쿼리 성능 정렬 최적화
* **ORDER BY 가속화**: 쿼리 작성 시 `ORDER BY 1 - (e.embedding <=> ?::vector) DESC`와 같이 사칙연산 수식을 적용하면, DB 옵티마이저가 정렬 인덱스를 효율적으로 타지 못하고 매 행마다 부하 연산을 수행합니다.
* **해결**: 인덱스 스캔 가속화를 위해 순수 코사인 거리 오름차순인 `ORDER BY e.embedding <=> ?::vector ASC`로 정렬하고, 유사도 스코어 계산(`1 - distance`)은 최종 결과 프로젝션을 위한 `SELECT` 절에서 단 1회만 계산되도록 최적화했습니다.

## 4. 실측 검증 및 레이턴시 분포 결과

프로젝트의 실질적 신뢰성을 담보하기 위해 **9개의 실제 PDF 문서**를 기반으로 **1,084개 청크**를 데이터베이스에 적재하고, **20개의 검증용 질문(골드셋)**을 구성하여 동작 성능을 수동 및 자동화 방식으로 정밀 평가했습니다.

### 📊 성능 계측 데이터 (평균 지연 시간)
20회 시나리오 루프를 기동하여 계측한 결과, 평균 지연 시간 분포는 다음과 같습니다.

```
RAG Total Response Time: 1,307ms (100.0%)
├── Retrieval (질문 임베딩 생성 + DB 유사도 검색)
│   └── 244ms (18.7%)
└── OpenAI Generation (Context 참고 gpt-4o 응답 생성)
    └── 1,063ms (81.3%)
```
> ※ 로컬 개발 환경 기준 측정값 (PostgreSQL 16 + Docker Compose + 개인 개발 PC)

* **병목 파악**: 병목의 약 81.3%가 외부 OpenAI API 네트워크 왕복 및 문장 생성 시간에 쏠려 있음을 확인했습니다.
* **백엔드 구간 성능**: 로컬 DB에서 유사도를 분석하고 임베딩을 요청하는 Retrieval 파트는 244ms 수준으로, 현재의 1,084개 청크 규모 하에서는 상대적으로 대단히 신속하게 처리되고 있음을 증명했습니다.

### 🧪 평가 방법론 및 정합성 검증 결과
* **Retrieval 적중성 수동 평가**: 20개 검증 질문에 대해 관련 문서 청크가 pgvector Top-5 검색 결과 안에 누락 없이 부합하는지를 직접 교차 검증하여 확인했습니다.
* **Groundedness(답변 정합성) 수동 평가**: 20개 질문셋 중, 적재된 문서에 정답이 없는 결측 질문에 대해서 LLM이 무작정 거짓말을 지어내지(환각 답변) 않고, 사전에 정의한 우회 메시지를 올바르게 출력하는지 직접 확인했습니다. 문서 내 정보 기반의 질문에 대해서는 컨텍스트 내용을 바탕으로 왜곡 없는 정밀한 답변이 유도됨을 입증했습니다.

## 5. 핵심 트러블슈팅 사례 (Troubleshooting Deep Dive)

### 1️⃣ TokenTextSplitter 설정 조율을 통한 한글 의미 유실 해결
* **문제 상황**: Spring AI `TokenTextSplitter`를 디폴트 설정으로 동작시켰을 때 한글 문장의 어절이나 의미 경계가 쪼개져 조사/어미가 잘리거나, 단어의 소실이 발생해 벡터 공간 상의 코사인 유사도 검색 정확도가 떨어지는 현상이 발생했습니다.
* **해결**: `TokenTextSplitter` 생성자에 문장 경계 보존 옵션인 `keepSeparator=true`를 적용하고, 청크 사이즈 파라미터를 **Chunk Size: 500 Tokens, Overlap: 100 Characters**로 상향 조절했습니다. 이를 통해 한글 문장의 끊김 현상을 개선하고 관련 문맥의 연속성을 효과적으로 확보했습니다.

### 2️⃣ PostgreSQL @Lob OID 매핑 오류로 인한 텍스트 깨짐 장애 해결
* **문제 상황**: 데이터베이스 적재 후 검색된 청크의 원본 텍스트를 불러오면 한글 텍스트 대신 `16912` 등 알 수 없는 정수(참조 포인터)만 출력되어 RAG 컨텍스트가 훼손되고 LLM이 답변을 거부하는 현상이 발생했습니다.
* **원인**: Hibernate Entity 설계 중 `content` 필드에 `@Lob` 어노테이션을 선언함에 따라, PostgreSQL 드라이버가 이 필드를 `TEXT`가 아닌 **Large Object (OID)** 데이터 타입으로 정의하여 바인딩했습니다. 이로 인해 DB 테이블에는 텍스트가 아닌 OID 고유 참조 번호가 들어갔고, JPA와 JDBC 간의 타입 불일치로 디코딩에 실패했습니다.
* **해결**: 엔티티 소스 코드에서 `@Lob` 어노테이션을 제거하고 `@Column(columnDefinition = "TEXT", nullable = false)` DDL 정의를 선언하여 실제 원본 문자열이 데이터베이스에 가공 없이 영속되도록 조치했습니다.

### 3️⃣ Spring AI AssistantMessage 호환성 문제 해결
* **문제 상황**: Spring AI 1.0.0-M6 마일스톤 버전 하에서 시스템 행동 지침(`SystemMessage`)과 질의 문맥(`UserMessage`)을 각각 독립 객체로 배열에 담아 `ChatModel.call(Prompt)`로 전송할 경우, 내부 직렬화 매핑 오류 혹은 `AssistantMessage` 바인딩 에러가 발생해 API 호출이 빈번히 거절되거나 문맥이 누락되었습니다.
* **해결**: 지시사항(Instruction), 지식 정보(Context), 사용자 질문(Question)을 하나의 템플릿 변수로 주입해 처리하는 단일 Combined Prompt 템플릿(`COMBINED_TEMPLATE`) 구조로 전송 프로세스를 리팩토링하여 프레임워크 수준의 바인딩 충돌을 방지하고 호환성 및 API 전송 안정성을 개선했습니다.

## 6. 기술 면접 대비용 꼬리 질문 (약점) 방어 가이드

### 💬 Q1. JPA의 변경 감지(영속 상태)와 JdbcTemplate 네이티브 SQL을 혼용했는데, 이로 인해 발생할 수 있는 "1차 캐시 불일치" 등의 정합성 문제는 어떻게 방어했는가?
* **답변 가이드**: 
  > "데이터 적재 및 임베딩 생성 시점은 단일 트랜잭션(`@Transactional`) 범위 내에서 보장됩니다. JPA 엔티티(`Document`, `Chunk`)를 데이터베이스에 완전히 `save()`하여 영속화한 뒤, 변경된 데이터를 DB 저장소에 반영하기 위해 명시적으로 `flush()` 처리하거나 트랜잭션 커밋 라이프사이클을 보장했습니다. `JdbcTemplate` 기반의 `EmbeddingStore`는 JPA를 우회해 데이터베이스의 데이터를 물리적으로 바인딩하므로, 조회 전 JPA 컨텍스트와 물리 DB 스키마 간 데이터 동기화 시점이 확실하게 맞아떨어지도록 코드 실행 순서를 제어했습니다."

### 💬 Q2. 현재는 청크 수가 약 1,000개 정도라 빠르지만, 만약 적재량이 10만 개 이상으로 대규모로 늘어난다면 검색 성능 하락을 어떻게 방어할 것인가?
* **답변 가이드**: 
  > "현재는 데이터 크기가 작아 pgvector 인덱스 없이 코사인 거리 연산을 통한 전수 조사(Flat Scan) 방식을 사용하고 있습니다. 데이터가 스케일 아웃되어 수만 개 이상의 대용량으로 늘어나면 응답 시간이 선형 증가하므로, pgvector가 지원하는 근사 최근접 이웃(ANN) 인덱스인 **HNSW(Hierarchical Navigable Small World)**를 활성화할 계획입니다. HNSW 인덱스는 메모리 사용량은 늘어나지만, 탐색 범위(m, ef_construction 등) 튜닝을 통해 재현율(Recall) 손실을 1~2% 내외로 방어하면서 지연 시간(Latency)을 O(log N) 수준으로 크게 단축시킬 수 있습니다."

### 💬 Q3. RAG 파이프라인에서 왜 LangChain 같은 최신 AI Agent 프레임워크나 복잡한 Workflow 도구들을 사용하지 않았는가?
* **답변 가이드**: 
  > "주니어 개발자가 엔지니어링 신뢰성이 확보되지 않은 상태에서 복잡한 AI Agent 프레임워크(예: LangChain ReAct Agent)나 도구를 성급히 적용하면 할루시네이션(환각) 확률이 증폭되고, 데이터의 동기화 흐름을 예측하거나 디버깅하기가 매우 까다로워집니다. 
  > 따라서 프레임워크의 화려한 기능에 의존하는 대신, RAG의 기본이 되는 데이터 파싱 정합성, 한글 청킹 기법 튜닝, pgvector 쿼리 정밀 제어, 구간별 레이턴시 계측 및 Fast-path Bypass 예외 처리에 집중하여 시스템의 기본 구조를 견고히 하는 것이 비즈니스와 엔지니어링 관점 모두에서 ROI(투자 대비 효과)가 가장 높다고 판단했습니다."
